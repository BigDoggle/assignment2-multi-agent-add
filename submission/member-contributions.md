# 成员贡献与反思素材

## 成员 1

- 中文姓名：王景宣
- 负责工作：主导实现、环境调通、运行验证、最终交付整理。
- 遇到的问题：本地默认 JDK 版本不满足项目运行要求，真实模型运行链路较长，输出材料较多且需要进一步整理成可提交成果。
- 解决方式：切换到 Homebrew Java 和 Maven 运行环境，逐步完成 smoke test 与正式运行验证，并将原始日志和报告草稿拆分整理为源码、日志、报告三类正式交付物。
- 个人反思：这次作业中最大的收获是把“模型调用”转化成“可复现的软件体系结构设计流程”。仅仅让模型产出内容不够，真正重要的是把流程、证据和最终文档组织成完整成果。

## 成员 2

- 中文姓名：李顺
- 负责工作：需求梳理、提示词约束检查、开发联调、结果检查、报告整理。
- 遇到的问题：多智能体输出内容存在重复和冗长现象，初始草稿难以直接映射到课程报告模板；同时需要确保系统没有引入题目之外的外部知识。
- 解决方式：围绕四轮 iteration 和 ADD Step 结构重新整理输出内容，协助比对 conversation log 与 report draft，筛选出可以直接进入最终报告的核心结论与图示。
- 个人反思：开发相关工作不仅是写代码，也包括对提示策略、输出结构和结果可用性的把控。通过参与联调和整理，我更理解了多智能体系统中“过程约束”对结果质量的重要性。

## 成员 3

- 中文姓名：张岩
- 负责工作：测试验证、日志与报告复核、提交目录整理、交付一致性检查。
- 遇到的问题：项目最终成果不只是代码，还包括日志、报告和提交材料，文件之间的关系较多，容易出现遗漏；另外，运行输出和最终提交版本之间需要做进一步区分。
- 解决方式：从交付视角重新核对源码、日志和报告产物，协助确认哪些文件属于正式证据、哪些属于中间草稿，并帮助完善最终提交清单与文档一致性检查。
- 个人反思：这次参与开发相关工作的过程中，我体会到软件作业的完成标准并不只是“程序能跑”，还包括验证、说明和可交付性。测试和复核工作虽然不显眼，但对最终成果质量很关键。
